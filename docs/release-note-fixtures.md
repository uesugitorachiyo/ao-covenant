# AO Covenant Release Note Fixtures

Use these fixtures as stable public examples for drafting and reviewing AO
Covenant release notes. They demonstrate normal releases, asset replacements,
withdrawals, and security-sensitive public summaries without exposing private
material.

Use this document with the [release note template](release-note-template.md),
[public release known-good baseline](public-release-known-good-baseline.md),
[release verification walkthrough](release-verification.md), and
[release rollback runbook](release-rollback.md).

## Scope

The fixtures are markdown examples for public release text. They are not
generated by the release workflow and must not contain real private reports,
private keys, credentials, production evidence, unreleased bundles, or local
machine paths.

## Fixture Inventory

- [normal.md](../internal/cli/testdata/release-note-fixtures/normal.md)
  covers an ordinary release where existing users do not need corrective action.
- [replacement.md](../internal/cli/testdata/release-note-fixtures/replacement.md)
  covers guarded replacement of already-published assets.
- [withdrawal.md](../internal/cli/testdata/release-note-fixtures/withdrawal.md)
  covers a release that should no longer be installed.
- [security-sensitive.md](../internal/cli/testdata/release-note-fixtures/security-sensitive.md)
  covers public wording for a security-sensitive fix without exploit details.

## Stable Content Requirements

Every fixture must include:

- affected version
- who is affected
- required consumer action
- what to download
- verification commands for release verification, release report generation,
  and GitHub artifact attestation
- an explicit sensitive-material exclusion

Replacement fixtures must mention `release-replacement-policy.json` and
`covenant.release-replacement-policy.v1`.

## Safety Requirements

Fixtures must use synthetic versions, asset names, timestamps, and summaries.
Do not include private keys, credentials, production evidence bundles,
unreleased bundles, local machine paths, exploit payloads, secret values,
customer data, private advisory details, tokens, or passwords.

## Maintenance

When the release note template or known-good baseline changes, update the
fixtures in the same pull request and run:

```sh
go test -count=1 ./internal/cli -run TestReleaseNoteFixturesAreLinkedAndComplete
```
